## 1920x1080 full console annotations

### `images/04_Pro_4decks.jpg`
- **Panel hierarchy:** top waveform rail → four-deck transport strips → center mixer core → lower browser strip.
- **Control grouping pattern:** mirrored deck clusters (transport, jog, loop, cue) around a centralized EQ/crossfader block.
- **Color semantics:** deck A family uses cool cyan/blue accents, deck B family uses warm red/magenta accents, utilities use amber/green.

### `images/05_Performance.jpg`
- **Panel hierarchy:** full-width waveform focus at top, dual-deck body, condensed browser/footer modules.
- **Control grouping pattern:** transport and beat controls are proximal; FX and pads are second-order groups beneath waveform context.
- **Color semantics:** left/right deck hues remain separated for rapid scanability, global statuses use neutral + amber warnings.

### `images/06_PerformanceFX.jpg`
- **Panel hierarchy:** waveform + deck core with expanded FX layer in the mid-band.
- **Control grouping pattern:** FX banks are columnar and mode-based, with per-deck send/intensity controls.
- **Color semantics:** deck channels retain A/B color separation; active FX state uses brighter glow overlays.

### `images/06_PerformanceFX (1).jpg`
- **Panel hierarchy:** same as FX-focused performance layout, with a stronger emphasis on effect module framing.
- **Control grouping pattern:** grouped FX modules are framed as independent lanes to reduce accidental cross-deck edits.
- **Color semantics:** deck colors still differentiate ownership; utility toggles use yellow/green state markers.

### `images/07_Vertical_waveforms.jpg`
- **Panel hierarchy:** dominant vertical waveform lanes, secondary deck transport below/adjacent.
- **Control grouping pattern:** cue/loop/sync controls are placed near each waveform lane for immediate correction.
- **Color semantics:** per-deck waveform lane tinting is the primary semantic channel; utilities remain neutral/amber.

### `images/08_video_mix.jpg`
- **Panel hierarchy:** video preview/output rails at top with audio deck/mixer controls below.
- **Control grouping pattern:** video routing/preview controls are isolated from transport to prevent mode confusion.
- **Color semantics:** deck A/B colors remain for audio ownership; video/broadcast utilities use green/amber emphasis.

### `images/08_video_mix_rack.jpg`
- **Panel hierarchy:** rack-style video module over standard deck/mixer substrate.
- **Control grouping pattern:** rack modules are tabbed/slot-based, with explicit enable-state indicators.
- **Color semantics:** A/B deck colors persist; rack status uses utility palette and high-contrast active states.

### `images/EssentialsMarked.png`
- **Panel hierarchy:** reduced “essential” mode keeps primary transport and browse modules visible.
- **Control grouping pattern:** flattened groups prioritize load/play/cue over deep FX operations.
- **Color semantics:** deck identification remains color-led while noncritical chrome is desaturated.

### `images/ProMarked.png`
- **Panel hierarchy:** pro mode exposes maximal panels: waveform, deck, mixer, browser, utility lanes.
- **Control grouping pattern:** strict modular framing with dense controls and predictable deck symmetry.
- **Color semantics:** robust A/B separation, with utility and warnings in green/amber/red channels.

### `images/StarterMarked.png`
- **Panel hierarchy:** starter mode compresses depth into guided primary zones.
- **Control grouping pattern:** beginner-safe grouping keeps transport and browsing central; advanced controls are deprioritized.
- **Color semantics:** deck colors stay consistent for transfer learning across starter/pro layouts.

## Do not copy (aligned with `docs/ui/dj_console_ui_research_plan.md`)

- Do not reuse vendor logos, brand names, or trademarked product labels from references.
- Do not replicate proprietary icon packs, glyph silhouettes, or branded pictograms one-to-one.
- Do not reproduce exact panel skins, texture maps, button sprites, or pixel-identical control chrome.
- Do not import proprietary layout preset names as first-party DGN-DJ feature labels.
- Do emulate: spatial hierarchy, control density, deck symmetry, and information rhythm while preserving **DGN-DJ by DGNradio** branding.

## Design extraction checklist (for future contributors)

1. Confirm file is inside `images/` and add exactly one table row per file.
2. Set `asset_type` to `screenshot` or `video`.
3. Record `resolution` in `WIDTHxHEIGHT` format for images; use `n/a` for video/non-image captures.
4. Assign `ui_area` from: `deck`, `mixer`, `waveform`, `browser`, `FX`, `sampler`, `routing`, `status`.
5. Fill `style_tags` with all six facets: density, contrast, glow, typography, spacing, module framing.
6. Fill `functional_tags` using only: transport, cue, loop, sync, browse, monitoring, automation.
7. Set `priority`: `P0` for core full-console references, `P1` for feature references, `P2` for ancillary/supporting captures.
8. If image is `1920x1080` full-console, add/update full-console annotations (panel hierarchy, control grouping, color semantics).
9. If image is `800x500` feature card, add/update feature annotation table (behavior + `src/components/` mapping).
10. Re-check “Do not copy” constraints before implementation work starts.
